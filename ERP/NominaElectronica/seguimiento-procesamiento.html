0La regla que gobierna todo el componente

⭐
El panel es un LECTOR, nunca un calculador. Todo lo que pinta sale de UNA sola fuente: la respuesta del endpoint ConsultarEstadoProcesoNomina(idProceso), que a su vez lee la tabla BERP_NominaElectronicaProceso. El panel no suma, no estima, no anima porcentajes por su cuenta, no tiene lógica de negocio. Si un dato no viene en la respuesta, no se muestra. Prohibido inventar avance (la pantalla vieja usaba Math.random() — eso es exactamente lo que se elimina).

④

⑥

Zona de resultado — solo en estados finales o interrupciones; contiene el mensaje y la acción primaria (ver §4).

2El contrato de datos (una sola respuesta alimenta todo)

GET/POST NominaElectronica/ConsultarEstadoProcesoNomina?idProceso=34

  "idProceso": 34,
  "estado": "EN_PROCESO",          // ENVIANDO | EN_PROCESO | COMPLETADO | CON_NOVEDADES | RECHAZADO | FALLIDO
  "proveedor": "DISPAPELES",       // DATAICO | DISPAPELES
  "jobId": null,                   // string (Dataico) o null (Dispapeles)
  "reportaParciales": true,        // decide la forma de la barra (§3)
  "total": 1012,
  "aceptados": 455,                // null si reportaParciales == false y no ha terminado
  "rechazados": 5,
  "pendientes": 552,
  "fechaCreacion": "2026-07-16 09:41",
  "fechaFinalizacion": null,       // set solo en estado final
  "mensajeError": null,            // set en FALLIDO / RECHAZADO
  "workerActivo": true             // solo Dispapeles: heartbeat vigente (§5)
}

Regla de oro del backend detrás de este endpoint (Task 7): si el proceso está EN_PROCESO con proveedor Dataico, primero reconcilia contra job-status/batch-result y actualiza la tabla; si es Dispapeles, responde directo de la tabla (el worker ya la mantiene al día). Si el estado ya es final, responde de BD sin llamar a nadie. Idempotente: llamarlo N veces no duplica nada.

5El ciclo de actualización (polling) — pseudocódigo del front

// Se entra al panel por 3 puertas — las 3 convergen en la misma función:
//   a) recién confirmado el envío (IniciarProcesamientoNomina devuelve idProceso)
//   b) al consultar un período que ya tiene proceso (ConsultarProcesoPorPeriodo)
//   c) al abrir un proceso desde el historial de Estado Proceso

function montarPanel(idProceso):
    detenerPollingAnterior()                  // clearInterval SIEMPRE antes de crear otro
    refrescar(idProceso)                      // primera pintada inmediata
    timer = setInterval(() => refrescar(idProceso), 15000)

function refrescar(idProceso):
    r = await ConsultarEstadoProcesoNomina(idProceso)
    pintarStepper(r); pintarBarra(r); pintarMeta(r); pintarResultado(r)   // §1-§4
    if (r.estado es final):
        detenerPolling()                       // no se consulta más: la BD ya es definitiva

// Higiene obligatoria:
//  - clearInterval al cambiar de tab, al consultar otro período y al llegar a estado final
//  - un solo timer vivo a la vez (guardar el handle en variable del módulo)
//  - si la petición falla (red), NO romper el panel: mantener lo último pintado
//    y reintentar en el próximo tick

6Quién escribe el avance (contexto que el panel NO implementa, pero debe respetar)

El panel es idéntico para ambos proveedores. La diferencia está detrás del endpoint:

DATAICO (lote):
  Panel ──15s──► Endpoint ──"¿cómo va job_X?"──► Dataico
                    │◄─────────respuesta──────────┘
                    └─ actualiza tabla ─► responde al panel

DISPAPELES (individual):
  Worker del servidor (QBWI): envía doc a doc en tandas,
  anota detalle por documento y contadores por tanda      ─► tabla
  Panel ──15s──► Endpoint ──lee la tabla──► responde   (no llama a Dispapeles)

Consecuencia para el front: no hay ningún "modo Dispapeles" en el JS del panel. El mismo código pinta ambos; las diferencias llegan en los datos (jobId null, reportaParciales, workerActivo).

Los dos personajes (no confundirlos)

La pregunta natural es: "¿mando una cédula, Dispapeles responde, la barra sube, mando la siguiente?". Casi — pero hay dos actores separados a propósito:

  • El repartidor (worker): llama al API de Dispapeles cédula por cédula y anota en la BD. No sabe que existe una pantalla.
  • La pantalla (panel): lee la BD cada 15 segundos y pinta la barra. No sabe cómo se envía nada.

La barra no sube "por la respuesta de Dispapeles" — sube porque la respuesta quedó anotada en la BD y la pantalla la leyó en su siguiente vistazo. Esa indirección es la que permite cerrar el navegador sin matar nada.

Las tres precisiones sobre el modelo "una cédula a la vez"

  1. ¿Quién llama al API? El worker del servidor — no la pantalla ni el operador. A las 09:43 el operador puede cerrar todo e irse: el cronómetro de arriba sigue corriendo igual.
  2. ¿La barra sube por cada respuesta? Cada cédula se anota al instante (su fila en el detalle), pero la barra avanza a saltos de tanda (0% → 8% → 17%…): los contadores se consolidan cada ~85 y la pantalla mira cada 15 s. Nadie necesita ver avanzar 0,09% por cédula — y ese desacople es lo que hace el sistema robusto.
  3. ¿Estrictamente de a una? No: 5 en vuelo a la vez (mientras Dispapeles procesa la cc#1, ya salieron la #2 a la #5). Mismo orden de ideas, 5 veces más rápido: ~5 minutos en vez de ~25. El grado de paralelismo y el tamaño de tanda son configurables (calibrar con los tiempos reales de Dispapeles — pregunta d de OPS-2).

9Checklist de aceptación del componente

  • Todo dato visible proviene de ConsultarEstadoProcesoNomina; no existe ningún cálculo de avance en el JS.
  • El paso 2 se pinta verde únicamente cuando la respuesta trae estado final.
  • Con reportaParciales:false jamás se muestra un número de porcentaje.
  • La barra siempre muestra la fracción junto al % ("45% · 460/1.012") cuando hay parciales.
  • El polling arranca en las 3 puertas de entrada, corre cada 15 s, y se detiene en estado final y al salir del contexto (sin timers huérfanos).
  • Cerrar y reabrir el navegador a mitad de un proceso muestra el estado correcto sin acción del usuario.
  • Los 7 estados de la máquina (§4) renderizan como se especifica — verificables sembrando registros en la tabla.
  • Caída de red transitoria no rompe el panel ni dispara alertas.
