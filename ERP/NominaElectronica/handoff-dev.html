1Lo esencial en 60 segundos

Qué construimos: el flujo de nómina electrónica deja de perder el resultado de las transmisiones. Cada procesamiento queda registrado en una tabla con su ciclo de vida (ENVIANDO → EN_PROCESO → COMPLETADO/CON_NOVEDADES/RECHAZADO/FALLIDO), el operador elige proveedor (Dataico o Dispapeles), el envío corre en segundo plano, la pestaña "Estado Proceso" pasa de mock a historial real, y los rechazados se corrigen y retransmiten individualmente como reemplazo.

La decisión de arquitectura central: ninguna pantalla calcula nada — todo el estado vive en BD (tablas BERP_NominaElectronicaProceso + Detalle) y las pantallas solo la leen por polling. Quien escribe el estado depende del proveedor: Dataico procesa en sus servidores (nosotros reconciliamos con su job-status/batch-result, que ya existen en DataicoHelper.cs y nunca se han usado); Dispapeles se envía documento a documento, así que el bucle corre como worker en segundo plano dentro del ERP — el tema técnico central de este handoff.

3La solución técnica del worker (léela antes de objetar 🙂)

Por qué un worker y no otra cosa

AlternativaPor qué se descartó
Todo en una petición~1.012 llamadas a Dispapeles ≈ 25 min → executionTimeout de ASP.NET (~110 s) la aborta; además secuestra hilos de IIS.
Bucle de promesas/tandas desde el navegadorResuelve el timeout pero el motor queda en la pestaña: cerrarla mata el envío. Requisito de negocio: el envío continúa sin operador.
Task.Run / hilo sueltoInvisible para ASP.NET: un recycle del app pool lo mata sin aviso, a mitad de un documento.
Hangfire / servicio WindowsVálidos pero sobredimensionados: dependencia o infraestructura nueva para un caso que nuestra tabla de procesos ya persiste. Si tienes argumentos fuertes a favor, conversemos — no es dogma.
✔ QueueBackgroundWorkItem (elegida)API oficial de ASP.NET (4.5.2+): registra el trabajo ante el runtime; en un recycle recibe CancellationToken y período de gracia (~30-90 s) para cerrar la tanda limpio. Cero infraestructura nueva.

Por qué el recycle NO es el argumento en contra (la clave del diseño)

El problema histórico de los hilos en IIS nunca fue el hilo — fue el trabajo no-reanudable. Aquí el trabajo es reanudable e idempotente por diseño:

  • Checkpoint por documento: cada cédula pasa por ENVIANDO_DOC (marca de intención, ANTES del POST) → envío → resultado anotado. Un crash entre enviar y anotar deja rastro: la reanudación verifica ese Nit en vez de reenviarlo → cero duplicados ante la DIAN.
  • Heartbeat por tanda: el worker actualiza una marca de tiempo con cada tanda. EN_PROCESO + latido vencido = worker muerto o congelado.
  • Auto-reanudación: cualquier poll que detecte latido vencido relanza el worker (QueueBackgroundWorkItem de nuevo) desde el primer pendiente. Botón "Reanudar" solo como respaldo manual.
  • Candado: el worker "toma" el proceso con un UPDATE condicional (solo si nadie con latido vigente lo tiene) → imposible que corran dos sobre el mismo proceso.

Código de referencia (~40 líneas — validar contra el repo en /implementar)

// DISPARADOR (en el controller — la petición responde en ~2 s):
int idProceso = NominaElectronicaProcesoRepository.CrearProceso(form, usuario);
HostingEnvironment.QueueBackgroundWorkItem(ct => DispapelesEnvioWorker.Ejecutar(idProceso, ct));
return Json(new { status = "success", idProceso });

// EL WORKER:
public static class DispapelesEnvioWorker
{
    public static void Ejecutar(int idProceso, CancellationToken ct)
    {
        try
        {
            if (!Repo.TomarProceso(idProceso)) return;          // candado (UPDATE condicional)
            var proveedor = new DispapelesProveedor();           // token auto-renovable adentro

            while (!ct.IsCancellationRequested)                  // recycle → salir limpio, SIN cerrar el proceso
            {
                var tanda = Repo.TomarPendientes(idProceso, 85); // excluye Nits ya registrados
                if (tanda.Count == 0) { Repo.CerrarProceso(idProceso); return; }

                Parallel.ForEach(tanda,
                    new ParallelOptions { MaxDegreeOfParallelism = 5, CancellationToken = ct },
                    emp =>
                    {
                        Repo.MarcarEnviando(idProceso, emp.Nit);              // 1. intención
                        var r = proveedor.EnviarDocumento(emp);               // 2. POST + respuesta (CON timeout)
                        Repo.GuardarResultado(idProceso, emp.Nit, r);         // 3. resultado
                    });

                Repo.ActualizarContadoresYLatido(idProceso);     // resumen por tanda → mueve la barra
            }
        }
        catch (Exception ex)
        {
            LogHelper.WriteLogV1(...);                           // jamás morir en silencio
            Repo.RegistrarErrorProceso(idProceso, ex.Message);
        }
    }
}

Checklist técnico que te pedimos validar (tu terreno, no el nuestro)

  • Configuración del app pool en producción: ¿idle timeout (default 20 min)? ¿recycle programado (default 29 h) a qué hora? Impacta la probabilidad de interrupciones.
  • Timeout HTTP por llamada a Dispapeles (RestSharp): obligatorio (~30 s) — sin él, un Dispapeles colgado congela el worker sin matarlo (el peor caso: vivo pero quieto).
  • MaxDegreeOfParallelism y tamaño de tanda: propuestos 5 y 85 — calíbralos con los tiempos reales y el rate limit de Dispapeles (tú los conoces).
  • El candado: valida que el UPDATE condicional del "TomarProceso" sea atómico en esta BD (nivel de aislamiento).
  • Vigilante opcional (OPS-7 del plan): tarea programada de Windows que llame cada 5 min un endpoint VerificarProcesosHuerfanos — detecta latidos vencidos sin depender de que alguien mire, y mantiene el app pool despierto. Recomendado; tu llamada.

4La abstracción que une los dos proveedores

interface IProveedorNominaElectronica {
    RespuestaEnvioNe Enviar(...);          // Dataico: lote completo → jobid · Dispapeles: lo usa el worker por documento
    EstadoLoteNe ConsultarEstado(...);     // Dataico: GET job-status · Dispapeles: lee nuestra BD (el worker ya la mantiene)
    ResultadoLoteNe ConsultarResultado(...);// Dataico: GET batch-result · Dispapeles: ya está en el detalle
    RespuestaEnvioNe Reemplazar(...);      // novedades: retransmitir un documento corregido
    bool ReportaAvanceParcial { get; }     // decide la forma de la barra (% real vs etapas)
}

Consecuencia clave: el front no tiene "modo Dispapeles" — el mismo panel pinta ambos proveedores; las diferencias viajan en los datos del único endpoint que alimenta todo el seguimiento:

// ConsultarEstadoProcesoNomina(idProceso) — el panel es un LECTOR de esta respuesta, nunca calcula:
{
  "idProceso": 34,
  "estado": "EN_PROCESO",     // ENVIANDO | EN_PROCESO | COMPLETADO | CON_NOVEDADES | RECHAZADO | FALLIDO
  "proveedor": "DISPAPELES",
  "jobId": null,              // string (Dataico) · null (Dispapeles, envío individual)
  "reportaParciales": true,   // true → barra con % real y fracción · false → etapas sin número
  "total": 1012, "aceptados": 455, "rechazados": 5, "pendientes": 552,
  "fechaCreacion": "…", "fechaFinalizacion": null, "mensajeError": null,
  "workerActivo": true        // solo Dispapeles: heartbeat vigente. false → panel muestra "interrumpido
}                             //   — se reanuda automáticamente" + botón Reanudar (respaldo)

Reglas del backend detrás: si es Dataico EN_PROCESO → reconcilia con job-status/batch-result y actualiza BD; si es Dispapeles → lee BD directo (el worker la mantiene); estado final → responde de BD sin llamar a nadie. Idempotente. El polling del front: cada 15 s con pantalla abierta, se detiene en estado final, clearInterval al cambiar de contexto.
